# 에이전트 작업 지침 (AGENTS.md)

## IMPORTANT !!
- 이 사항들이 안 지켜질 경우, 사용자가 험한 욕설을 할 것임 !
- 코드 수정이 끝나고 커밋을 했는데, main 실행 후 에러가 발생하면 절대로 안된다.
- GUI 기능 수정은 반드시 e2e 테스트로 검증해야 됨.
- 검증의 가치가 없는 테스트 코드를 검증하는 행위는 자아도취 자기만족 탁상공론 임.

## README.md 관리 규칙 (CRITICAL)
`README.md`를 업데이트하거나 수정할 때, 다음 섹션들은 시스템 구조와 기술적 상세 파악을 위해 필수적인 정보이므로, 사용자의 명시적인 삭제 요청이 없는 한 **절대로 삭제하거나 요약하여 생략하지 마십시오.**
- **프로젝트 구조**
- **주요 진입점**: 소스 코드의 파일별 역할 및 진입 경로
- **실행 중 프로세스 구성**: 고루틴별 역할 및 비고
- **웹 서버 엔드포인트**: API 경로, 메서드, 상세 설명
- **데이터 플로우**: ASCII 다이어그램
- **설치 및 실행 (Quick Guide)**

## 문서 관리 규칙
1. **폴더 구조 및 목적**: `doc/` 폴더 하위의 문서는 다음 체계를 따른다.
    - `ing/`: 현재 진행 중인 작업 관련 문서
    - `hold/`: 대기 중이거나 보류된 작업 관련 문서
    - `done/`: 완료된 작업의 기록 문서
    - `discard/`: 폐기된 작업 문서
2. **이동 원칙**: 에이전트는 문서를 임의로 폴더 간 이동시키지 않는다. 사용자의 명시적인 요청이 있을 때만 수행한다.
3. **신규 생성**: 신규 버그 리포트나 작업 문서는 기본적으로 `ing/` 또는 사용자가 지정한 폴더에 생성한다.

## 명명 규칙 (Naming Conventions)
- **소스 파일 및 폴더**: 파일명, 함수명, 변수명 등 영어를 사용하며, Go 표준 관례를 따른다. (camelCase/PascalCase)
- **소스 파일 예외**: 주석은 모두 한국어를 사용하며, 로그 메시지 및 앱 내 문자열 상수는 한국어를 사용한다.
- **테스트 파일**: 소스 파일과 같은 폴더에 `_test.go` 접미사로 작성한다. 별도 `tests/` 폴더를 만들지 않는다.
- **문서 및 기타 파일**: 가급적 한국어를 사용한다.
- **커밋 메시지**: 한국어를 사용한다.
- **커밋 명령어**: `git commit -m "메시지"` 형식을 사용한다. `cat` heredoc 방식은 사용하지 않는다.

## 기술적 제약
- macOS 전용 앱으로, CGo를 통해 macOS 프레임워크를 직접 호출할 수 있다.
- TDD(테스트 주도 개발) 방식을 따르며, Go 표준 `testing` 패키지를 활용한다.
- CGo가 포함된 코드는 `_darwin.go` / `_darwin.c` 파일 접미사를 사용하여 플랫폼을 명시한다.

## 대응 전략 (YOLO 모드)
- **Pre-emptive Execution**: "작업을 하겠다"는 말과 함께 즉시 첫 번째 tool call을 실행하고, 의존성이 없는 경우 연속적으로 실행한다.
- **Chain of Thoughts in Action**: 예상되는 시나리오를 바탕으로 연속적인 행동 계획을 세우고 연쇄적으로 실행한다.
- **Stopping Criteria**: 사용자의 확인이 필요한 시점에만 명확히 멈춘다.

## 작업 문서 규칙

### 파일 생성 규칙
- 작업 1개당 파일 1개를 `doc/` 하위 적절한 폴더에 생성한다.
- **일반 작업**: `todo_작업명.md` (예: `todo_듀얼세이브_구현.md`)
- **버그 수정**: `bug_버그명.md` (예: `bug_클립보드_복사_실패.md`)

### 작업 결과 기록 규칙
- 작업이 완료되면 해당 파일 **하단에 별도 섹션**으로 결과를 추가한다.
